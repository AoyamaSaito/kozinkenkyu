# tools/ ── 生成スクリプト（B層）

デザインはシナリオの世界観に依存するため、スクリプト本体は毎回作り直す。
ただし以下の設計原則は全シナリオ共通で**必ず守る**。

## 設計原則

1. **レンダラは本文を持たない**: 配布物に載る文言（カード本文・キャラシート文言・論文テキスト等）を
   スクリプト内にハードコードしない。必ず `data/` のファイルから読み込む。
   - 理由: 正史⇔スクリプト内文言の二重管理は同期漏れ事故の温床。
     『覗きの代』ではキャラシート目標文言の同期漏れ（E-2）がテストプレイまで発覚しなかった
2. **1成果物群=1データファイル**: 「証拠カードP1ぶん」「キャラシート1人ぶん」の単位で
   `data/` 配下にファイルを分ける。文言修正＝データファイル編集＋再生成1回で完結させる
3. **パスはスクリプト位置基準の相対パス**: ドライブ直書き（`D:\...`）禁止。
   `os.path.dirname(os.path.abspath(__file__))` 起点で解決する（複数環境で作業するため）
4. **dist/ は手動編集禁止**: 配布物の修正は必ず data/ 編集 → 再生成の経路で行う
5. **エンコーディング**: 全ファイル UTF-8。フォントパスは環境依存なので定数として先頭に集約する

## 参考実装（『覗きの代』: `D:\個人研究その2\murder_mystery\udonarium_test\`）

- `generate_gm_pdf.py` — markdown（正史）→ PDF 変換。正史を直接読むため二重管理が発生しない**良例**
- `design_card.py` / `design_charsheet.py` — PIL 描画の作り込み例。描画技術は流用価値が高いが、
  **文言のハードコードは反面教師**（原則1違反の実例）
- `build_udon.py` — Udonarium 用 zip + XML ビルダー
