# tools/ ── 生成スクリプト（B層）

デザインはシナリオの世界観に依存するため、スクリプト本体は毎回作り直す。
ただし以下の設計原則は全シナリオ共通で**必ず守る**。

## 設計原則

1. **レンダラは本文を持たない**: 配布物に載る文言（カード本文・キャラシート文言・論文テキスト等）を
   スクリプト内にハードコードしない。必ず `data/` のファイルから読み込む。
   - 理由: 正史⇔スクリプト内文言の二重管理は同期漏れ事故の温床。
     『覗きの代』ではキャラシート目標文言の同期漏れ（E-2）がテストプレイまで発覚しなかった
2. **1成果物群=1データファイル**: 「証拠カードP1ぶん」「キャラシート1人ぶん」の単位で
   `data/` 配下にファイルを分ける。文言修正＝データファイル編集＋再生成1回で完結させる
3. **パスはスクリプト位置基準の相対パス**: ドライブ直書き（`D:\...`）禁止。
   `os.path.dirname(os.path.abspath(__file__))` 起点で解決する（複数環境で作業するため）
4. **dist/ は手動編集禁止**: 配布物の修正は必ず data/ 編集 → 再生成の経路で行う
5. **エンコーディング**: 全ファイル UTF-8。フォントパスは環境依存なので定数として先頭に集約する
6. **1成果物=1スクリプト（.py）**（2026-07-13 ユーザー指示・00_DECISIONS.md 記録済）:
   生成物ひとつにつき、それを出力するスクリプトをひとつ置く。
   例外は証拠カード等の**同型量産物のカテゴリ単位まとめ**のみ。
   「全部入りビルドスクリプト」は作らない——壊れたとき・差し替えたいときの影響範囲を1成果物で閉じる

## このシナリオの生成方式（マダミス4）

**全成果物で以下の「三点セット」構成を標準とする**（2026-07-13 ユーザー確定）:

| 役割 | 場所 | 触るとき |
|---|---|---|
| 本文の正 | `data/<distと同じ階層>/<成果物名>.md` | 文章を直すとき（普段はここだけ） |
| 見た目 | `tools/templates/<成果物名>.html` | スタイル・レイアウトを変えるとき |
| 生成 | `tools/render_<成果物名>.py` | 基本触らない（実行するだけ） |

data/ の階層は **dist/ をミラー**する（例: `dist/配布物/共通/導入シート.png` ← `data/配布物/共通/intro_sheet.md`。詳細: data/README.md）

パイプライン: **MD（本文の正）→ テンプレートに流し込み → Playwright（システムの Edge/Chrome を channel 起動）→ PNG撮影**

- デザイン言語の仕様: `scenario/07_visual_design.md`（2077 UI文法×赤の時代の大気）
- 判型: 横 1600×1000（16:10）、出力は device_scale_factor=2 の 3200×2000
- HTML は Artifact プレビューと共用のため doctype 無しの素片。スクリプト側でフルドキュメントに包んで撮影する
- 前作の PIL 直描き方式は不採用: 今作のネオン系デザイン（グラデ・グロー・グリッチ・字間制御）は CSS の方が再現・修正コストが低い
