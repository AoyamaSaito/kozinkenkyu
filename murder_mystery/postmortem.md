## 再現すべきパターン

- **推理ロジックはキャラの後**: キャラの動機・スキル・行動が先にないと手がかり配置が宙に浮く
- **全体フロー→個別差分の二層**: ②をフルで書くと矛盾の温床。全体を先に確定し、個別は差分のみ重ねる
- **AIシミュレーションは構造の煙感知器**: 情報分離の機械的検証（HO・カードの突き合わせでネタバレが漏れないか）には有効。ただしAI PLは「理想的すぎるプレイヤー」であり、実卓の主な事故（見落とし・忘却・誤読）は再現できない。**体験の検証は実卓のみ**。4人ワンショット規模なら構造は脳内でも追えるため、シミュは省略可能な工程
- **ジャンル調査を先にやる**: いきなり作らず、ジャンルの文法を押さえてから設計に入る
- **廃案を恐れない**: コンセプト初稿が合わなければ同日中に捨てて方向転換
- **格子状証拠構成（消去法格子）**: 複数物証の交点で犯人を一意特定する構造は、設計者にとって検証しやすく、PLに達成感がある。PC4変更時の整合性検証が半日で済んだのはこの明快さゆえ

## 反省点・改善議論（2026-07-03 実施）

### ユーザー側の反省点

1. **4人構成の役回りフォーマット**: 「犯人／疑われる人／確信に近い情報を持つが推理が必要な人／ダントツで白い人」の4枠は一般的な構成。今回は世界観→キャストの順で決めて結果的にこのフォーマットに乗ったが、次回はフォーマットをテンプレートとして先に持ち、キャスト決定時に4枠への当てはめを**検証ステップ**として挟む（枠から作るのではなく、枠で検証する）
2. **世界観とギミック・推理要素はセットで考える**: 世界観・シナリオラインだけ先に決めるとギミック・推理の後付けが大変。設定確定時に推理の着地条件（「この構造なら消去法で犯人特定できるはず」レベル）を仮置きしておく三段構え（世界観＋ギミック骨格はセット／推理の着地条件は仮置き／格子の精緻化はキャラの後）
3. **犯人の情報量**: 犯人は自分の正体に繋がる証拠情報を多く持つべき（上から2番目以内）。妨害・嘘といった犯人らしいプレイングの原資になる。議論の結果、本質は情報の**量**ではなく「**証拠に対して犯人が取れるアクションの分岐（動詞）**」であると整理。前提として「全証拠が揃えば犯人に辿り着ける」保証設計＋「それを制限し抗う余地」が必須
4. **直近改修の検証**: 院生→ジャーナリスト変更＋査読口頭制限＋確保目標は、段階A（口頭制限＝歪める動詞）／段階B（行動隠蔽性）／段階C（出歩きのカバーストーリー）の全てでBの干渉手段を構造的に追加できていると確認。Bの売り先も罠①民俗学者・罠②調査隊員・罠③教授の3本が確保され、罠③は「教授が査読するほど疑惑が育つ」自動性が教授自身の勝利条件と直結しているため構造的に保証される

### AI側の指摘

1. **シミュレーションの限界**（→上記「再現すべきパターン」に反映済）
2. **情報の分散と変更伝播コスト**: PC4変更で20ファイル、Phase番号振り直しで9ファイル21箇所。ドキュメント間不整合2件も発生。原因は同じ事実の複数ファイル重複記載。→ 下記「次回の情報整理プラン」
3. **設計判断の根拠がセッション履歴にしか残っていない**: 却下案と却下理由（ホワイトアウト廃止・「密度で勝負」却下・揺らぎ却下）が成果物に残らず、postmortem作成にセッション発掘が必要だった。棄却案の再導入事故のリスク。→ 決定ログの常設（下記プラン参照）

## 次回の情報整理プラン（テンプレート案）

> 現行『覗きの代』のファイルは変更しない。次回作で最初からこの構造で始めるための案。

### レイヤー構造（今回の実績から抽出）

```
A. 設計ドキュメント（scenario/）    ← 手動編集。横の重複が不整合の温床だった
B. 生成スクリプト（tools/）         ← A→Bは手動反映。追跡リストで管理
C. 配布物（dist/）                  ← B→Cは自動生成。同期漏れなし（今回の成功点）
```

手動同期が必要な面は「A内部の横の重複」と「A→B」の2つだけ。対策もこの2点に絞る。

### A層のファイル構成案

```
scenario/
├── 00_DECISIONS.md      # 決定ログ（常設・1決定1行）
│                        #   例: 2026-06-16 | ホワイトアウト廃止 | 役割が見えない | 代替: 固有ターン×2
├── 01_concept.md        # 【正】世界観・キャスト表・スキル一覧・勝利条件
├── 02_flow.md           # 【正】Phase構成・全体ビートシート（キャスト詳細は01参照）
├── 03_deduction.md      # 【正】推理格子・手がかり配置・情報二層・罠
├── 04_characters/       # 差分のみ（全体フローの複製禁止。「→02参照」で書く）
├── 05_truth.md          # 真相編・完全時系列（GM用）
└── 09_CHANGES.md        # 変更追跡リスト（常設。今回の10_testplay_fixes.md形式を最初から）
```

### 運用ルール

1. **事実のオーナー制**: 各事実（キャスト名・スキル仕様・Phase構成…）の定義場所は1ファイルのみ。他ファイルは「→01参照」形式で参照し、コピペしない。配布物（HO・カード）は自己完結が必須なので重複可＝一方向化の対象は**設計ドキュメント間のみ**
2. **決定ログは決定の瞬間に書く**: 却下案＋却下理由＋採用案を1行。書くコストは10秒。判断が速い制作スタイルだからこそ根拠の外部化を機械的に行う
3. **変更追跡リストの常設**: 設計変更が発生したら都度 `09_CHANGES.md` に影響範囲（A→B→C）を列挙してから着手する。テストプレイ後ではなく設計期から
4. **共有データの機械可読化（検討）**: キャスト名・Phase名・日付・タイトル等を `scenario_data.json` に一元化し、B層スクリプトはそこから読む。PC4改名級の変更が「JSONの1箇所＋設計ドキュメント」で済むようになる
5. **役回りフォーマット検証**: キャスト決定時に「犯人／疑われ役／情報持ち推理役／白」の4枠に当てはめて過不足を確認する
6. **推理の着地条件を設定確定時に仮宣言**: 「設定A確定」に相当するタイミングで「この構造なら◯◯の消去法で犯人特定に至る」を1文で書いておく（格子の精緻化はキャラ後で可）
